粗略地看了一遍这本书，大多是讲算法的，当做算法的复习吧。

1 面试流程

1.1 面试形式

大体流程（三轮）：（简历筛选 - ）电话面试 - 远程面试 - 现场面试。

1.1.1 电话面试

  • 要求在投简历后的一两个星期内保持电话能持续通话一个小时。
  • 尽可能用形象的语言把细节说清楚。
  • 当不清楚面试官问题时，要向面试官提问至理解面试官意图为止。

1.1.2 远程面试

这种方式只有一些大公司会采用。在当中，面试官比较关心应聘者的编程习惯和调试能力。

  • 思考清楚再开始写代码。
  • 良好的代码命名合 缩进对齐习惯。
  • 能够单元测试。

1.1.3 现场面试

  • 规划好路线并估算出行时间。
  • 准备好得体的衣服（舒服干净即可）。
  • 注意面试邀请函里的面试流程。
  • 准备几个问题。

1.2 面试环节

每轮面试大概分为三个环节：行为面试 - 技术面试 - 应聘者提问。

1.2.1 行为面试

  • 时间：5-10分钟。
  • 流程：30s~60s的自我介绍 - 项目经验 - 掌握技能 - 为什么跳槽。
  • 项目经验（STAR模型：Situation - Task - Action - Result）
    • 关键字：参与，负责。
    • 简短的项目背景。
    • 自己完成的任务。
    • 为了完成任务自己做了哪些工作，是怎么做的。
    • 自己的贡献。
    • 其它问题
      • 你在该项目中碰到的最大的问题是什么，你是怎么解决的？
      • 从这个项目中你学到了什么？
      • 什么时候和其他团队成员有什么冲突，你们是怎么解决冲突的？
  • 掌握技能
    • 关键字：了解，熟悉，精通。
    • 了解：指对某一个技术只是上过课或看过书，并没有实际的项目。
    • 熟悉：使用某一项技术已经有较长的时间，通过查阅文档可以解决大部分问题。
    • 精通：可以很轻松地回答这个领域的绝大多数问题。
  • 为什么跳槽
    • 避免
      • 老板太苛责。
      • 同事难相处。
      • 加班太频繁。
      • 工资太低。
    • 合适
      • 缺少激情，想要挑战。

1.2.2 技术面试

  • 时间：40~50分钟。
  • 素质
    • 基础知识扎实全面，包括编程语言、数据结构、算法。
    • 能写出正确的、完整的高质量代码。
    • 能思路清晰地分析、解决复杂问题。
    • 能从时间、空间复杂度两方面优化算法效率。
    • 具备优秀的沟通能力、学习能力、发散思维能力等。

1.2.3 应聘者提问

  • 时间：5-10分钟。
  • 准备
    • 面试前做好功课，去网上收集信息。
    • 根据面试官说过的话提问。
  • 适合提问内容：与职位或者面试相关的。

1.3 自己将面对的面试（总结，后面有经验了会来更新）

1.3.1 提前准备

  • 简历制作
    • 样式简洁舒服
    • 语言简明扼要
  • 简历投递
    • 相关网站：拉勾网，实习僧

    • 一些字眼：如了解、熟悉、精通、参与、负责等等。
    • 对于项目的描述，注意分点，便于回答。（上面有写到）

1.3.2 面试流程

简历通过后便会进入正式面试环节，主要有基础面，项目面和最后的Hr面三部分。（每一个环节刚开始可能都会有个简单的自我介绍）

3 相关性能问题

大多数浏览器是使用单一线程（UI线程）来处理用户界面和执行js脚本的，因此，当js代码在执行时就可能出现页面渲染和用户交互完全被阻塞的问题（如形成白屏），下面是针对该类问题的一些解决方案。

  • 脚本位置
    • 将脚本放在body底部。
  • 组织脚本
    • 压缩js文件，减小文件大小。
    • 合并js文件，减少http请求。
  • 延迟脚本（延迟的时间不同）
    • defer：在script元素内设置，可以异步加载，但是到DOM树构建完成后才会执行js脚本。
    • window.onload：在js文件内部设置，到页面所有资源加载完毕后才会执行js脚本。
  • 异步脚本
    • async：异步加载，但不能确定js脚本的执行时间。

发表于 2017-01-15 |

1 版本控制是什么？

版本控制是一种记录一个或若干文件内容变化，以便将来查阅特定版本修订情况的系统，主要有集中式（CVCS）和分布式（DVCS）两种。

2 Git工作流程？

基本的Git工作流程如下：

  • 在工作目录中修改文件。
  • 暂存文件，将文件的快照放入暂存区域。
  • 提交更新，找到暂存区域的文件，将快照永久性存储到Git仓库目录中。

Git有三种状态，在这三个流程中Git分别处于已修改（modified）、已暂存（staged）和已提交（committed）。

  • 已修改（modified）：表示修改了文件，但还没保存到数据库中。
  • 已暂存（staged）：已暂存表示对一个已修改文件的当前版本做了标记，使之包含在下次提交的快照中。
  • 已提交（committed）：已提交表示数据已经安全的保存在本地数据库中。

另外，在这三个流程中有三个重要的区域：

  • 工作区域（Working Directory）：是对项目的某个版本独立提取出来的内容。 这些从 Git 仓库的压缩数据库中提取出来的文件，放在磁盘上供你使用或修改。
  • 暂存区域（Staging Area）：是一个文件，保存了下次将提交的文件列表信息，一般在 Git 仓库目录中。 有时候也被称作 索引 ，不过一般说法还是叫暂存区域。
  • Git仓库（Repository）：Git 用来保存项目的元数据和对象数据库的地方，从其它计算机克隆仓库时，拷贝的就是这里的数据。
